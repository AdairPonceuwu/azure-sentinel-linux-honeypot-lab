# Architecture Notes

## Overview

This lab uses a Debian 12 virtual machine in Microsoft Azure as an SSH honeypot. The VM is intentionally exposed to the public Internet through TCP/22 using an Azure Network Security Group rule.

SSH authentication activity generated by `sshd` is logged through Linux Syslog. Azure Monitor Agent collects those logs using a Data Collection Rule and forwards them to a Log Analytics Workspace. Microsoft Sentinel then analyzes the data using KQL and visualizes attacker source locations in a custom Workbook attack map.

## Architecture Flow

1. External hosts on the public Internet attempt to connect to the honeypot over SSH.
2. The Azure Network Security Group allows inbound TCP/22 traffic.
3. The Debian 12 VM receives the SSH connection attempts through OpenSSH Server.
4. SSH authentication activity is written to Linux Syslog.
5. Azure Monitor Agent collects Syslog authentication events.
6. A Data Collection Rule defines which Syslog facilities are collected.
7. Logs are forwarded to the Log Analytics Workspace.
8. Microsoft Sentinel queries the `Syslog` table using KQL.
9. Attacker IP addresses are extracted from `sshd` messages.
10. IP addresses are enriched with geolocation data using `geo_info_from_ip_address()`.
11. A Sentinel Workbook visualizes the results on an attack map.

## Main Components

| Component | Purpose |
|---|---|
| Public Internet / Attackers | Source of SSH authentication attempts |
| Azure Network Security Group | Allows inbound TCP/22 traffic to the honeypot |
| Debian 12 Honeypot VM | Linux server exposed for controlled monitoring |
| OpenSSH Server | Service receiving SSH authentication attempts |
| Linux Syslog | Local logging mechanism for SSH authentication events |
| Azure Monitor Agent | Collects logs from the Linux VM |
| Data Collection Rule | Defines which Syslog facilities and levels are collected |
| Log Analytics Workspace | Central log repository |
| Microsoft Sentinel | SIEM used for analysis and visualization |
| KQL Queries | Extract and summarize attacker IP activity |
| GeoIP Enrichment | Adds location context to attacker IPs |
| Sentinel Workbook | Displays the SSH attack map |

## Data Source

The main data source for this lab is the `Syslog` table.

Relevant SSH log patterns include:

- `Invalid user`
- `Connection closed by invalid user`
- `Connection reset by invalid user`
- `Failed password`
- `Accepted publickey`

## Data Collection Rule Configuration

The DCR was configured to collect Linux Syslog authentication events.

Recommended facilities:

| Facility | Minimum Level |
|---|---|
| LOG_AUTH | Info |
| LOG_AUTHPRIV | Info |

Other facilities can remain disabled unless additional Linux logs are needed in future phases.

## Design Notes

This lab was originally inspired by a Windows honeypot architecture that used Windows Security Events and Event ID `4625`. Because the Azure student subscription did not allow creating a Windows VM, the design was adapted to Debian Linux.

The final architecture uses:

- `Syslog` instead of `SecurityEvent`
- `sshd` authentication messages instead of Windows Event ID `4625`
- `geo_info_from_ip_address()` instead of a Sentinel Watchlist for GeoIP enrichment

## Security Notes

This VM is intentionally exposed for educational and defensive security purposes. It should not contain sensitive data and should remain isolated from production systems.

Before publishing screenshots, remove or censor:

- Public IP addresses
- Subscription IDs
- Tenant IDs
- Resource IDs
- Usernames
- SSH key names
